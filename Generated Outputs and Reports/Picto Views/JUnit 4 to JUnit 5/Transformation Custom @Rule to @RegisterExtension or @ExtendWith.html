Transformation: Custom @Rule to @RegisterExtension or @ExtendWith

Migration relation type: Transformation
Migration scope: JUnit 4 => JUnit 5 / JUnit Jupiter and JUnit Platform

Applicability conditions: None specified.

Transformation intent

Source feature: Custom TestRule
Target feature(s): @RegisterExtension, @ExtendWith, BeforeEachCallback, AfterEachCallback, ExecutionCondition, ParameterResolver
Automation: NoDirectMapping / COMPLEX, MIX
Relation: Custom JUnit 4 rules have no automatic general mapping. They must be classified according to the lif ecycle phase or service they implement and then redesigned as one or more Jupiter extensions.

Operators: Migration engineer / test developer

No-direct-mapping rationale: JUnit Jupiter does not natively support arbitrary JUnit 4 rules. Only selected rule types are suppor ted by migration-support; custom TestRule/MethodRule behavior may change execution flow and needs re design.

Possible approximation feature(s): @RegisterExtension, @ExtendWith, BeforeEachCallback, AfterEachCallback, ExecutionCondition, ParameterResolver
Approximation profile: COMPLEX, MIX

Realization strategy


1. [M] Classify the custom rule behavior: before/after resource, conditional disabling, exception

 handling, parameter injection, reporting, or statement wrapping.



2. [M] If it is a supported ExternalResource/Verifier/ExpectedException subclass, decide whether 

migration-support is acceptable temporarily.



3. [M] If the rule wraps Statement to alter control flow, manually design an equivalent extension

 or accept that no direct equivalent exists.



4. [M] Implement the closest Jupiter extension callback set.



5. [M] Register with @ExtendWith when no per-test instance configuration is needed; otherwise use

 @RegisterExtension.



6. [M] Validate behavior with passing, failing, skipped, and aborted tests.
