Transformation: JUnit 4 Suite Runner to JUnit Platform Suite

Migration relation type: Transformation
Migration scope: JUnit 4 => JUnit 5 / JUnit Jupiter and JUnit Platform

Applicability conditions: None specified.

Transformation intent

Source feature: @RunWith(Suite.class)
Target feature(s): @Suite, @SelectClasses, @SelectPackages, @SuiteDisplayName, junit-platform-suite-engine
Automation: OperatorHelp / MIX
Relation: JUnit 4 explicit suite classes are migrated to the JUnit Platform suite API. Several target states a re possible: direct selected classes, package selectors, tag filters, engine filters, or temporary V intage execution.

Operators: Migration engineer / test developer

Realization strategy


1. [A] Detect suite class annotated with @RunWith(Suite.class).



2. [A] Extract classes from @Suite.SuiteClasses.



3. [A] State 1 - direct class suite: create a suite class annotated with @Suite and @SelectClasse

s({ ... }).



4. [M] State 2 - package-based suite: if selected classes share stable packages and the source in

tent is package aggregation, use @SelectPackages instead of listing classes.



5. [M] State 3 - filtered platform suite: if the old suite was used to simulate categories/groups

, combine @Suite with @IncludeTags/@ExcludeTags or class-name patterns.



6. [M] State 4 - mixed-engine suite: if both Jupiter and Vintage tests remain, keep Vintage on ru

ntime path and optionally use @IncludeEngines/@ExcludeEngines.



7. [M] State 5 - transitional JUnitPlatform runner: when a JUnit 4 environment must run JUnit Pla

tform suites, use @RunWith(JUnitPlatform.class) only as a temporary compatibility state.



8. [A] Add junit-platform-suite or suite-api and suite-engine dependencies.



9. [M] Run the suite and compare discovered test count with the original JUnit 4 suite.
