Transformation: Compartment Mapping to Container Mapping

Migration relation type: Transformation
Migration scope: GMF Workbench => Sirius Workbench

Applicability conditions:

  • REQUIRED on Compartment Mapping: Apply when a GMF Compartment Mapping represents a nested visual region that hosts child elements.

Transformation intent

Source feature: Compartment Mapping
Target feature(s): Container Mapping
Automation: FullAutomation / EASY
Relation: A GMF Compartment Mapping and a Sirius Container Mapping serve the same core purpose: they define a nested graphical container inside a node, used to visually group and structurally contain child elem ents. Since Sirius does not have a separate compartment concept independent of containers, a Contain er Mapping is the most direct target feature that preserves the compartment's role as a host region with an internal hierarchy.

Realization strategy


1. [A] take the parent top node or child reference (A) of the compartment mapping



2. [A] take A's equivalent container mapping (B)



3. [A] create a new container mapping (C) as a child of B



4. [A] write 'aql:self' into the semantic candidates expression of C

Strategies of nested transformations

Computing Domain Class


1. [A] copy the element feature of A's equivalent node mapping and paste it into domain class

Computing Id


1. [A] take the figure (D) of the visual representation compartment feature



2. [A] copy the name of D and paste it into id
